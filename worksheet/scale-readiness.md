# 전사 확장 준비도 워크시트

> [문서 06](../docs/06-enterprise-scale.md)의 확장 국면 진화를 현장에서 점검하는 양식입니다. 확장은 더 큰 빅뱅이 아니므로, 이 워크시트는 "전사 청사진을 먼저 그렸는가"가 아니라 "검증된 수요가 다음 진화를 끌어당기고 있는가"를 봅니다. 빈칸이 많은 것은 채우라고 비워둔 것이지 미완성이 아닙니다.

- **누가** — AX 추진 리더 / 전담조직(CoE, Center of Excellence) 리더가 주관하고, 플랫폼·아키텍처 책임자(섹션 A·B), 거버넌스·리스크 담당과 재무/FinOps(섹션 C)가 자기 칸을 채웁니다. 확장/투자 판정은 임원(CIO·CDO·CAIO, 최고 AI 책임자)이 검토합니다.
- **언제** — 한 유스케이스의 확장이 아니라 **여러 파일럿에 걸친 공통 수요가 보이기 시작할 때**, 그리고 이후 분기마다 갱신합니다. 단일 파일럿의 확장/중단은 이 양식이 아니라 [파일럿 게이트](pilot-gate.md)에서 판정합니다.
- **채운 뒤** — 섹션 A의 신호가 임계에 닿은 계층만 [문서 03 §3.1.6](../docs/03-alternative.md)의 platform-pull 후보로 등록하고, 섹션 B의 결정 기록은 결재 근거이자 재검토 알람으로 보관하며, 섹션 C의 기준선은 [문서 05 §5.5](../docs/05-operating-model.md) 비례적 거버넌스와 [문서 08](../docs/08-roadmap.md)의 측정 지표로 연결합니다.

작성일: ______________   주관(추진/전담조직 리더): ______________   검토 임원: ______________
대상 범위(어느 도메인·부서군의 확장인가): ______________

> 입력 필수도 표기 — **[필수]** 이 판단 없이는 확장 결정을 내릴 수 없는 항목 · **[권장]** 근거를 튼튼히 하는 항목 · **[선택]** 해당 시에만. 처음 쓰는 용어는 첫 사용 시 한국어로 풀고 영문을 병기하며, 정착되면 [A1 용어집](../appendix/A1-appendix.md) 등록 대상입니다.

## A. 확장 전환 준비도 점검 (신호 체크)

확장은 선언이 아니라 신호로 당겨집니다. 아래 신호가 **여러 파일럿에 걸쳐 반복 관찰될 때만** 다음 계층을 끌어올립니다(pull). 신호가 없는데 미리 까는 것(push)은 [문서 01](../docs/01-diagnosis.md)이 비판하는 빅뱅으로의 회귀입니다.

### A-1. Platform-pull 신호 — 공통 수요가 반복되는가

각 계층에서 "같은 것을 반복해서 필요로 한 파일럿 수"를 셉니다. **출처**: [파일럿 게이트](pilot-gate.md) §4의 platform-pull 신호 칸을 계층별로 집계. **측정**: 같은 계층을 독립적으로 요구한 서로 다른 파일럿의 수(중복 요구 횟수가 아니라 파일럿 수).

| 공통 계층(후보) | 반복 요구한 파일럿 수 | 임계 도달 | 보류 사유(임계 도달했으나 미승격 시) |
|------------------|:---:|:---:|------|
| 모델 서빙·추론 엔드포인트 [필수] |  | ☐ |  |
| 사내 검색·RAG 파이프라인 [필수] |  | ☐ |  |
| 접근 통제·권한·시크릿 관리 [필수] |  | ☐ |  |
| 평가셋·품질 측정 공통 기반 [권장] |  | ☐ |  |
| 관측·로깅·추적(observability) [권장] |  | ☐ |  |
| 데이터 거버넌스·계보(lineage) [권장] |  | ☐ |  |
| 가드레일·안전 필터 공통화 [선택] |  | ☐ |  |

> 임계 기준(예시, 조직이 직접 정함): 서로 다른 2 – 3개 파일럿이 같은 계층을 독립적으로 요구하면 그 계층만 플랫폼 후보로 승격. 한 파일럿의 강한 요구는 임계가 아닙니다. '임계 도달'을 체크한 계층은 기본적으로 끌어올릴 후보입니다 — 도달했는데도 끌어올리지 않는 경우에만 마지막 칸에 보류 사유(예: 선행 공백 보강 필요·재검토 시점 대기)를 적습니다.

**판정 — 끌어올릴 계층(있으면):** ______________ · **연결 → [문서 07 Private AI](../docs/07-private-ai-strategy.md)** 도입 시점 검토(데이터 주권·규제·단가가 함께 임계면).

### A-2. 역량 플라이휠 성숙도 — 재사용이 실제로 일어나는가

[문서 03 §3.1.5](../docs/03-alternative.md)의 자산이 쌓이기만 하는지, 실제로 다음 파일럿을 싸게 만드는지 봅니다. **출처**: 자산 등록처([파일럿 게이트](pilot-gate.md) §3의 재사용 자산 등록처). **측정**: 등록된 자산을 재사용한 후속 파일럿 수, 신규 파일럿 착수 비용·기간의 추세.

- 등록된 재사용 자산 수(평가셋·파이프라인·가드레일·플랫폼 부품): ______ [필수]
- 그중 둘 이상 파일럿이 실제로 재사용한 자산 수: ______ [필수]
- 최근 파일럿 착수 비용·기간 추세(이전 대비 ↓ / 유지 / ↑): ______ [권장]
- 재사용이 안 일어나는 자산이 있다면 이유(발견 안 됨 / 품질 / 맥락 불일치): ______________ [선택]

> 재사용률이 낮으면 확장 신호가 아니라 **자산 발견·표준화의 공백** 신호입니다. 플랫폼을 더 까는 대신 등록처·표준을 먼저 손봅니다.

### A-3. 운영모델 진화 준비 — 중앙에서 연합으로 넘길 수 있는가

[문서 05 §5.1](../docs/05-operating-model.md)의 중앙집중 → 연합(federated) 진화가 인위적 조직 개편이 아니라 역량으로 당겨지는지 봅니다. **출처**: 현업 오너 배치 현황, 재교육 이수 현황. **측정**: 현업이 자력으로 끝낸 파일럿 비율.

- ☐ 현업 프로덕트 오너가 도메인마다 지정되어 있다 [필수]
- ☐ 현업이 중앙 도움 없이 한 사이클을 끝낸 사례가 있다 [필수]
- ☐ 중앙이 직접 수행에서 플랫폼·표준·코칭으로 무게를 옮길 준비가 됐다 [권장]
- ☐ 게이트·표준이 문서화되어 현업이 따라 할 수 있다 [권장]

> 위 항목이 비어 있으면 연합 모델로의 전환은 이릅니다. 권한 이양을 서두르면 [문서 01](../docs/01-diagnosis.md)의 변화관리 공백·거버넌스 약화가 동시에 옵니다.

### A-4. 비가역 결정의 임박도 — Two-way-door 점검

[문서 03 §3.1.4](../docs/03-alternative.md). 지금 검토 중인 확장 결정이 되돌릴 수 있는지부터 가립니다.

- 이번 분기 검토 중인 확장 결정: ______________
- 성격: ☐ 가역(two-way door) — 실험으로 빠르게 검증  ☐ 비가역(one-way door) — 증거 축적 후 결정
- 비가역이라면, 그 결정을 뒷받침하는 증거(A-1 – A-3 신호): ______________ [필수]

> 비가역 결정(전사 표준 확정·대규모 선투자·조직 개편)을 가역인 척 빠르게 내리는 것이 빅뱅 실패의 전형입니다. 증거가 얇으면 결정을 더 작게·가역적으로 쪼갭니다.

## B. 아키텍처 결정 기록 (진화형)

확장기의 아키텍처는 "선행 청사진"이 아니라 검증된 수요로 당겨지는 진화형 규율입니다. 결정 하나마다 한 블록을 채우고, **가역성과 재검토 시점을 반드시 적습니다**. 되돌릴 수 없는 결정일수록 재검토 알람을 더 촘촘히 둡니다. 이 양식은 흔히 ADR(아키텍처 결정 기록, Architecture Decision Record)로 불리며, 여기서는 가역성·재검토를 필수 칸으로 둔 진화형 변형입니다([A1 용어집](../appendix/A1-appendix.md) 등록 대상).

> 구체 인프라·제품 구현(서빙 엔진·벡터 DB·노드 사이징 등)은 이 워크시트에서 정하지 않습니다. [문서 07 Private AI](../docs/07-private-ai-strategy.md)와 외부 Private AI 시리즈로 위임합니다. 여기서는 "무엇을, 왜, 얼마나 되돌릴 수 있게" 결정했는지만 기록합니다.

## C. 거버넌스·FinOps·신뢰성 기준선

확장 국면에서는 파일럿 단위 점검만으로는 부족하고, 여러 유스케이스에 걸친 **기준선**이 필요합니다. 그러나 기준선도 한 번에 전사로 깔지 않고, 끌어올린 계층(섹션 A·B)에 비례해 세웁니다. 각 칸은 현재값과 목표·임계, 그리고 그 값이 어디로 연결되는지를 함께 적습니다.

> 진입 게이트 — 섹션 A-5에서 끌어올린 계층이 없으면(빈 신호가 정상입니다) C는 현재 운영 중인 유스케이스 범위에서 **[필수] 항목만** 채우고 나머지는 다음 분기로 미룹니다. 끌어올린 계층에 비례해 그 계층이 닿는 기준선부터 칸을 엽니다 — 측정 대상이 아직 없는 칸을 미리 채우지 않습니다.

> 출처·측정 공통 규칙 — 정량 수치(GPU·노드·용량·단가)가 필요하면 본 양식에서 계산하지 말고 [외부 사이징·비용 편](https://github.com/JaeHoYun/vcf-private-ai/tree/main/06-sizing-cost)의 워크북 결과를 이 표에 옮겨 적습니다. 본 워크시트에는 자동 계산이 없습니다.

### C-1. 거버넌스 기준선

[문서 05 §5.5](../docs/05-operating-model.md) 비례적 거버넌스를 확장 규모로 확장합니다. 등급별 통제를 게이트에 통합한다는 원칙은 그대로 두되, 다수 유스케이스에 일관 적용되는지를 봅니다.

|------|-----------|-----------|----------------|:---:|
| 위험 등급 분류가 모든 운영 유스케이스에 매겨졌나 |  |  | 미분류 건 → 게이트 재심 | [필수] |
| 고위험 유스케이스의 휴먼인더루프(HITL) 적용률 |  |  | 미달 → 통제 보강·승인 보류 | [필수] |
| 공통 가드레일(접근통제·로깅·평가·사고대응) 적용 범위 |  |  | 공백 도메인 → 우선 보강 | [필수] |
| 모델·프롬프트·데이터 변경 이력 추적 가능 여부 |  |  | [문서 08](../docs/08-roadmap.md) 감사 추적 | [권장] |

### C-2. FinOps 기준선

확장기에는 비용이 파일럿 상한이 아니라 **운영 단가**로 바뀝니다. 비용 가시성과 단위 경제가 임계를 넘으면 그 자체가 platform-pull·Private AI 전환 신호입니다([문서 07](../docs/07-private-ai-strategy.md)).

### C-3. 신뢰성 기준선

여러 부서가 의존하기 시작하면 가용성·품질이 더 이상 파일럿 수준일 수 없습니다. 다만 신뢰성 목표도 위험 등급에 비례합니다 — 저위험 내부 도구에 고가용성 기준을 강요하지 않습니다.

| 항목 | 현재값 | 목표·임계 | 다음 활용/연계 | 필수도 |

| 등급별 가용성 목표(SLO)와 실적 |  |  | 미달 → 운영 보강·범위 축소 | [필수] |

| 의존도 집중(단일 모델·단일 공급자 쏠림) |  |  | 높으면 → 대체 경로 two-way 확보 | [권장] |

## 채운 뒤 어떻게 활용하나

- **보관·갱신** — 팀 내부에 두고 분기마다, 그리고 큰 확장 결정 직전에 갱신합니다.
- **의사결정 근거** — 섹션 A의 신호와 섹션 B의 결정 기록은 확장·투자 결재의 근거이자 사후 감사 추적입니다.
- **환류** — 섹션 C에서 드러난 임계는 다시 섹션 A의 platform-pull·[문서 07 Private AI](../docs/07-private-ai-strategy.md) 전환 신호로 돌아갑니다.
- **경영진 보고** — 신호 추세·결정 이력·기준선 변화는 [문서 08](../docs/08-roadmap.md)의 측정 지표로 그대로 보고에 씁니다. 활동량(허영 지표)이 아니라 검증된 수요와 결과로 보고합니다.
