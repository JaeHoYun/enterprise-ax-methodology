# 거버넌스·데이터주권 갭

> 이 워크시트는 이미 AI를 광범위하게 쓰는 조직이 **현존 AI 자산의 거버넌스·데이터 주권 통제가 어디까지 갖춰졌는지**를 표준 프레임에 대고 실측하는 역방향(기존 자산 감사·정리) 양식입니다. 같은 폴더의 [확장 준비도](scale-readiness.md)가 신규 운영 확장을 앞두고 "다음 진화를 끌어당기는가"를 보는 전방 양식이라면, 이 양식은 반대로 **이미 깔려 운영 중인 것들의 통제 격차**를 되짚습니다. 격차를 메우는 일도 "전사 거버넌스 청사진을 먼저 그려라"가 아닙니다 — 실측한 격차와 위험에 비례해, 실제로 검증된 필요가 우선순위를 끌어당기는 순서로 메웁니다. 빈칸이 많은 것은 채워 넣으라고 비워 둔 것이지, 양식이 미완성이라는 뜻이 아닙니다.

- **누가** — AX 추진 리더 또는 거버넌스·리스크 책임자가 주관하고, 보안·정보보호(개인정보) 담당이 데이터 주권 칸을, 법무·컴플라이언스가 규제 프레임 칸을, 플랫폼·운영 담당이 자산대장 실측 칸을 채웁니다. 판정은 임원(최고정보책임자 CIO·최고데이터책임자 CDO·최고AI책임자 CAIO 또는 정보보호최고책임자)이 검토합니다.
- **언제** — 브라운필드 진단 착수 시 1회 전수 점검하고, 이후 분기마다 그리고 규제 데이터를 다루는 신규 자산이 운영에 들어갈 때 갱신합니다. 자산이 흩어져 있을수록 한 번에 완성하려 하지 말고 운영 중인 것부터 우선 채웁니다.
- **채운 뒤** — 채운 값은 종합 개선안 보고서의 **S6(거버넌스·주권 갭) 절** 입력이 되고, 식별된 고위험 격차는 **전환 프로젝트 차터의 리스크 항목**으로 옮겨 적습니다. 구체 구현(데이터 온프렘 회귀·마이그레이션·통제 자동화 실행)은 이 양식에서 정하지 않고 [문서 07 Private AI](../docs/07-private-ai-strategy.md)와 외부 Private AI 시리즈로 위임합니다. 진단·격차 식별 자체는 특정 사설 AI 플랫폼(예: Private AI Foundation, PAIF) 채택 여부와 무관하게 성립합니다.

작성일: ______________   주관(거버넌스·리스크 리더): ______________   검토 임원: ______________
대상 범위(어느 도메인·법인·데이터 등급까지 점검하는가): ______________

> 입력 필수도 표기 — **[필수]** 이 점검 없이는 격차·위험을 보고서·차터로 넘길 수 없는 항목 · **[권장]** 근거를 튼튼히 하는 항목 · **[선택]** 해당 시에만. 인용하는 표준 프레임은 격차를 비추는 거울일 뿐이며, 특정 벤더 제품·인증 취득 자체가 목적이 아닙니다. 처음 쓰는 용어는 첫 사용 시 한국어로 풀고 영문을 병기하며, 정착되면 [A1 용어집](../appendix/A1-appendix.md) 등록 대상입니다.

## A. AI 자산대장 (실측 계량)

격차 판정에 앞서 **무엇이 운영되고 있는지부터 실측**합니다. 이 대장은 성숙도 자가진단의 레벨 라벨이 아니라 계량값을 담으며, 거꾸로 [성숙도 자가진단](maturity-self-assessment.md)의 거버넌스·리스크 차원 레벨을 판정할 때 그 증거 입력이 됩니다(자산대장 → 레벨, 한 방향 참조). 등록되지 않은 채 운영되는 자산(섀도 AI)을 드러내는 것이 이 절의 핵심 목적입니다([문서 06 §6.6](../docs/06-enterprise-scale.md)의 상시 가시성).

**출처**: 클라우드 결제·API 키 발급 내역, 베어메탈 GPU 스케줄러 작업 로그, 사내 앱·에이전트 배포 목록, 부서별 자가 신고. **측정**: 운영 중(스테이징·실험 제외) 기능 단위로 한 행씩. 한 행이라도 비면 "미상"으로 적고 추적 과제로 남깁니다.

|---------|-----------|-----------|---------------------------------------------|----------------------------------------|:---:|:---:|:---:|

|  |  |  |  |  |  | ☐ | [필수] |

|  |  |  |  |  |  | ☐ | [권장] |
|  |  |  |  |  |  | ☐ | [선택] |

> 행이 늘면 위 표를 이어 붙입니다. 위험 등급은 [문서 05 §5.5.1](../docs/05-operating-model.md)의 저·중·고 기준을 그대로 씁니다. **"등록 여부" 칸에 ☐(미등재 운영)가 많을수록 섀도 AI가 넓다는 신호**이며, 그 자체가 보고서 S6의 핵심 격차입니다.

## B. 표준 프레임 대비 통제 격차

운영 중 자산 전체(또는 위험 상위 자산)를 대상으로, 표준 프레임의 통제 의도에 견주어 현 격차를 채웁니다. 프레임은 거울일 뿐이며, 모든 항목을 한꺼번에 메우라는 뜻이 아니라 **위험에 비례해 메울 순서를 가리는 입력**입니다.

> 출처·측정 공통 규칙 — "현 통제 상태"는 실제 운영 증거(설정·로그·문서)로 적고 희망값을 적지 않습니다. "격차"는 의도와 현 상태의 차이를 한 줄로, "다음 활용/연계"는 그 격차가 보고서·차터·다른 워크시트 중 어디로 가는지를 적습니다.

### B-1. NIST AI RMF 대비 통제 격차

미국 표준기술연구소의 AI 위험관리 프레임워크(NIST AI Risk Management Framework)의 네 기능 — 거버넌스 수립(Govern), 맥락·위험 식별(Map), 측정(Measure), 대응·관리(Manage) — 에 대고 현 통제가 어디까지 와 있는지 봅니다.

| 기능 | 통제 의도(요약) | 현 통제 상태 | 격차 | 다음 활용/연계 | 필수도 |
|------|-----------------|--------------|------|----------------|:---:|
| Govern (거버넌스) | AI 위험을 다루는 정책·역할·책임이 정해져 있는가 |  |  | 공백 → 차터 거버넌스 항목 | [필수] |
| Map (맥락·위험 식별) | 각 자산의 용도·맥락·이해관계자 위험이 식별됐는가 |  |  | A절 자산대장과 대조 | [필수] |
| Measure (측정) | 위험·성능·편향을 측정하는 평가가 있는가 |  |  | [확장 준비도 C-3](scale-readiness.md) 신뢰성 | [권장] |
| Manage (관리) | 식별·측정된 위험에 대응·완화·모니터링이 도는가 |  |  | 고위험 미대응 → 차터 리스크 | [필수] |

### B-2. EU AI Act 고위험 기술문서(Annex IV) 대비 준비도

유럽연합 인공지능법(EU AI Act)은 고위험으로 분류되는 AI 시스템에 부속서 IV(Annex IV) 기술문서를 요구합니다. **해당 규제 시장에 서비스·제품을 제공하거나 EU 거주자 데이터를 처리하는 자산에 한해** 채웁니다(비해당이면 그 자산은 "해당 없음"으로 표기하고 근거를 남깁니다).

| Annex IV 요구 항목(요약) | 현 준비 상태 | 격차 | 다음 활용/연계 | 필수도 |
|--------------------------|--------------|------|----------------|:---:|
| 시스템의 일반 설명·의도된 목적 |  |  | 미작성 → 차터 산출물 과제 | [필수] |
| 개발·설계 선택과 시스템 구성 요소 |  |  | 아키텍처 결정 기록과 연계 | [권장] |
| 학습·검증 데이터의 출처·특성·관리 |  |  | C절 데이터 주권 점검과 대조 | [필수] |
| 성능·정확도·견고성·사이버보안 지표 |  |  | B-1 Measure와 동일 증거 | [권장] |
| 위험관리 체계와 사후 모니터링 계획 |  |  | B-1 Manage와 연계 | [필수] |
| 해당 자산이 고위험 분류 대상인지 판정 |  |  | 미판정 → 법무 검토 의뢰 | [필수] |

### B-3. ISO 42001 운영 체계 대비 격차

ISO/IEC 42001은 AI 경영시스템(AI Management System) 표준으로, 통제를 일회성 점검이 아니라 **돌아가는 체계**로 갖췄는지를 봅니다. 인증 취득이 목적이 아니라 체계 공백을 비추는 거울로 씁니다.

| 체계 요소 | 현 상태 | 격차 | 다음 활용/연계 | 필수도 |
|-----------|---------|------|----------------|:---:|
| AI 정책·목표가 문서화되고 책임자가 지정됐나 |  |  | B-1 Govern과 중복 점검 | [권장] |
| 자산·위험을 주기적으로 재평가하는 절차가 있나 |  |  | 분기 갱신 주기와 연계 | [권장] |
| 공급자·외부 모델에 대한 통제(제3자 위험)가 있나 |  |  | A절 외부 API 자산과 대조 | [필수] |
| 사고·시정조치를 기록·환류하는 고리가 있나 |  |  | [문서 05 §5.5.2](../docs/05-operating-model.md) 사고 대응 | [권장] |

> 세 프레임은 통제 의도가 겹칩니다. 같은 격차가 여러 표에 잡히면 중복으로 세지 말고 **가장 위험이 큰 자산 기준 한 건의 개선 과제**로 묶어 보고서로 넘깁니다.

## C. 데이터 주권 소급 점검

규제 데이터가 과거에 **외부 인프라(퍼블릭 클라우드·외부 API·국외 리전)를 경유한 적이 있는지**를 소급해 되짚습니다. 앞으로의 통제가 아니라 이미 일어난 데이터 흐름을 추적하는 것이 핵심이며, 여기서 드러난 사실은 온프렘 회귀·내재화 결정의 근거가 됩니다([문서 07 Private AI](../docs/07-private-ai-strategy.md)로 위임).

**출처**: API 호출 로그·네트워크 트래픽 기록, 데이터 처리 위탁·국외 이전 계약, 클라우드 리전 설정, 외부 모델 제공자의 데이터 사용·보존 약관. **측정**: 규제 데이터를 처리하는 A절 자산마다 한 줄씩. 로그가 없어 확인 불가하면 "확인 불가"로 적고 그 자체를 위험으로 다룹니다.

| 점검 항목 | 확인 결과 | 근거(로그/계약/약관) | 다음 활용/연계 | 필수도 |
|-----------|-----------|----------------------|----------------|:---:|
| 규제 데이터가 외부 API·퍼블릭 클라우드로 전송된 적이 있는가 | ☐ 있음 ☐ 없음 ☐ 확인 불가 |  | "있음/불가" → 차터 최우선 리스크 | [필수] |
| 전송된 데이터가 외부 모델 학습·로그에 보존되었을 가능성 | ☐ 가능 ☐ 차단됨 ☐ 확인 불가 |  | "가능/불가" → 약관·계약 재검토 | [필수] |
| 국외 리전을 경유했는가(데이터 거주지 위반 소지) | ☐ 예 ☐ 아니오 ☐ 확인 불가 |  | "예/불가" → 법무 검토 의뢰 | [필수] |
| 위탁·국외 이전에 필요한 동의·계약 근거가 갖춰졌나 | ☐ 갖춤 ☐ 미비 ☐ 확인 불가 |  | "미비/불가" → 컴플라이언스 과제 | [권장] |
| 외부 경유 데이터를 온프렘 회귀·삭제할 경로가 있는가 | ☐ 있음 ☐ 없음 ☐ 확인 불가 |  | → [문서 07](../docs/07-private-ai-strategy.md) 내재화 검토 | [권장] |

> "확인 불가"는 무해한 빈칸이 아니라 **소급 추적성의 부재**라는 위험 신호입니다. 로그·계약으로 입증되지 않는 외부 경유는 일어난 것으로 간주하고 차터 리스크로 올립니다. 다만 이 발견이 곧장 "전부 온프렘 회귀해 사내로 옮겨라"를 뜻하지는 않습니다 — 온프렘 회귀·통합·폐기는 위험과 검증된 필요에 비례해 [문서 07](../docs/07-private-ai-strategy.md)에서 순서를 정합니다.

## 채운 뒤 어떻게 활용하나

- **보관·갱신** — 팀 내부에 두고 분기마다, 그리고 규제 데이터를 다루는 신규 자산이 운영에 들어갈 때 갱신합니다.
- **보고서 입력** — A절 실측·B절 격차·C절 소급 결과는 종합 개선안 보고서의 S6(거버넌스·주권 갭) 절로 그대로 옮깁니다. 활동량이 아니라 실측 격차로 보고합니다.
- **차터 입력** — D절에서 추린 고위험 격차는 전환 프로젝트 차터의 리스크 항목이 되고, 온프렘 회귀·내재화가 필요한 건은 [문서 07 Private AI](../docs/07-private-ai-strategy.md)로 위임합니다.
- **레벨 판정의 증거** — A절 자산대장은 [성숙도 자가진단](maturity-self-assessment.md)의 거버넌스·리스크 차원 레벨을 매길 때 증거 입력이 됩니다.

## 중요

- 이 워크시트는 **채워넣기 양식**입니다. 항목별로 직접 값을 적고 판단을 기록하는 용도이며, 자동 계산은 없습니다. 정량 수치(온프렘 회귀 용량·내재화 비용 등)가 필요하면 외부 사이징·비용 시리즈([⑥ vcf-private-ai-sizing-cost](https://github.com/JaeHoYun/vcf-private-ai-sizing-cost)) 워크북 결과를 옮겨 적습니다.
- 인용한 표준 프레임은 격차를 비추는 거울이며, 특정 벤더 제품 채택이나 인증 취득 자체가 목적이 아닙니다. 진단은 벤더 중립입니다.
- 어떤 칸도 "전사 거버넌스 청사진을 먼저 완성하라"로 읽지 않습니다. 격차는 위험과 검증된 필요에 비례해 메웁니다.
- 모든 항목은 작성 시점 기준이며 환경에 따라 달라집니다. 적용 전 [상위 README](../README.md)와 본문으로 재확인하세요.
