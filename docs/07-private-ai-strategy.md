# 07 — Private AI로 AX 갖추기

> 두 번째 핵심 문서입니다. [문서 03](03-alternative.md)의 점진적 AX를 실제 사내 AI 토대 위에서 갖추는 전략을 다룹니다.

점진적 AX의 파일럿이 검증되고 확장되기 시작하면, 거의 반드시 한 질문에 부딪칩니다. "이 데이터를, 이 규모로, 외부 서비스에 계속 보내도 되는가." 이 지점에서 Private AI(사내에서, 데이터가 밖으로 나가지 않은 채 운영하는 AI)가 전략의 중심으로 들어옵니다.

이 문서는 Private AI가 왜 필요해지는지, 언제 전환하는지, 무엇으로 구성되는지, 그리고 그것을 다시 빅뱅으로 만들지 않으려면 어떻게 해야 하는지를 다룹니다. 구체 구현은 벤더 중립적 개념으로 설명하고, 한 가지 구체 경로로 PAIF/VCF 기반 시리즈를 참조로 연결합니다.

---

## 7.1 왜 Private AI인가

검증된 유스케이스를 실제 업무에 올릴 단계가 되면 다음 압력이 동시에 커집니다.

- **데이터 주권** — 사내 지식·고객 정보·영업 자료가 모델 입력으로 들어갑니다. 이 데이터가 조직 경계를 벗어나는 것이 허용되지 않는 경우가 많습니다.
- **규제·컴플라이언스** — 개인정보·산업별 규제·내부 통제가 외부 전송을 제약합니다.
- **지식재산 보호** — 핵심 노하우가 외부 서비스의 학습·로그에 남는 위험을 피해야 합니다.
- **비용 단가** — 사용량이 임계를 넘으면 외부 API 종량 과금보다 사내 운영이 단가에서 유리해지는 구간이 옵니다.
- **지연·가용성** — 업무에 깊이 결합될수록 응답 지연과 가용성을 자체 통제할 필요가 커집니다.
- **벤더 종속 회피** — 특정 외부 서비스에 핵심 업무가 묶이는 위험을 분산합니다.

핵심은 이 압력들이 **처음부터가 아니라 검증·확장 단계에서 커진다**는 점입니다. 그래서 Private AI는 점진적 AX의 자연스러운 다음 단계이지, 전제 조건이 아닙니다.

이 전환이 개별 조직의 사정을 넘어 업계 전반의 흐름이라는 외부 시장 신호도 같은 방향을 가리킵니다(출처·수치는 [A1.3 참고](../appendix/A1-appendix.md)).

## 7.2 퍼블릭에서 시작할 것인가, 처음부터 프라이빗인가

모든 파일럿을 처음부터 사내 인프라로 시작할 필요는 없습니다. 다만 데이터 민감도에 따라 출발점이 갈립니다.

| 조건 | 권장 출발점 |
|------|-------------|
| 민감 데이터 없음, 빠른 검증이 목적 | 외부 API·샌드박스로 빠르게 파일럿 |
| 민감 데이터 일부, 가역적 실험 | 익명화·합성 데이터로 외부에서 검증 |
| 규제·기밀로 외부 전송 불가 | 처음부터 사내(Private) 환경에서 파일럿 |
| 검증 완료, 실데이터·실규모 운영 | Private AI로 전환 |

판단 기준은 두 가지입니다. **데이터가 나가도 되는가**(주권·규제), 그리고 **결정이 가역적인가**([문서 03](03-alternative.md)의 two-way-door). 외부에서의 빠른 검증은 가역적 실험이고, 실데이터·실규모 운영은 비가역에 가까우므로 사내 토대를 전제로 합니다.

## 7.3 단계적 전환 경로

점진적 AX와 Private AI 도입을 하나의 경로로 겹쳐 놓으면 다음과 같습니다.

```
[파일럿]            [확장 판정]              [운영]
외부 API 또는    ── 게이트 통과 +  ──   Private AI로 전환
사내 샌드박스        임계 초과 신호         (실데이터·실규모)
민감도 낮게 검증     · 데이터 민감도            │
                    · 사용량·단가             ▼
                    · 규제 요구          공통 수요 누적 시
                    · 지연·가용성        사내 플랫폼으로 흡수
```

전환을 촉발하는 임계 신호는 다음과 같습니다.

- 파일럿이 실데이터·민감정보를 요구하기 시작함
- 사용량이 늘어 외부 종량 과금이 사내 운영 단가를 넘어섬
- 규제·감사 요구가 외부 전송을 막음
- 업무 결합도가 높아져 지연·가용성을 자체 통제해야 함

이 신호 중 하나라도 확실해지면, 해당 유스케이스를 Private AI로 옮겨야 합니다. 신호가 없는 파일럿까지 미리 옮기지는 않아야 합니다.

## 7.4 Private AI 빌딩블록

Private AI는 단일 제품이 아니라 여러 계층의 조합입니다. 벤더 중립적으로 보면 다음으로 구성됩니다.

| 계층 | 역할 | 시리즈 참조 |
|------|------|-------------|
| 컴퓨트(GPU) | 추론·학습을 위한 가속 자원 | [⑥ 사이징·비용](https://github.com/JaeHoYun/vcf-private-ai-sizing-cost) |
| 프라이빗 클라우드 플랫폼 | 컴퓨트·스토리지·네트워크·쿠버네티스 기반 | [① 인프라·운영](https://github.com/JaeHoYun/vcf-private-ai-guide) |
| 모델 서빙 | 모델을 API로 제공 | [③ 서빙 API](https://github.com/JaeHoYun/vcf-paif-serving-api-guide) |
| 데이터·검색(RAG) | 사내 지식을 모델에 연결 | [② 벡터 DB](https://github.com/JaeHoYun/vcf-dsm-vectordb-guide) · [④ RAG 아키텍처](https://github.com/JaeHoYun/vcf-rag-reference-architecture) |
| 에이전트·도구 연동 | 모델이 사내 시스템과 일하게 함 | [① 인프라·운영](https://github.com/JaeHoYun/vcf-private-ai-guide) |
| 보안·거버넌스 | 격리·접근통제·감사·데이터 거버넌스 | [⑤ 보안·거버넌스](https://github.com/JaeHoYun/vcf-private-ai-security-governance) |

이 계층들을 하나의 통합 플랫폼으로 제공하는 구체적 경로의 한 예가 VMware Cloud Foundation 기반 Private AI(PAIF/PAIS)이며, 위 시리즈가 인프라부터 보안·비용까지 계층별로 다룹니다. 전체 구성은 [VCF Private AI 가이드 시리즈 허브](https://github.com/JaeHoYun/vcf-private-ai-series)에서 확인할 수 있습니다. 방법론 관점에서 중요한 것은 특정 제품이 아니라, **이 계층들을 한꺼번에 다 갖추려 들지 말고 수요가 증명된 순서로 채워 가는 것**입니다.

## 7.5 Platform-pull과 Private AI 도입 타이밍

[문서 03](03-alternative.md)의 platform-pull 원칙이 Private AI 도입 타이밍을 결정합니다.

- 첫 파일럿은 최소 구성으로 시작해야 합니다. 서빙 한 가지, 데이터 한 묶음.
- 둘째·셋째 파일럿이 같은 서빙·검색·접근통제를 반복 요구하면, 그 공통분모가 플랫폼화 신호입니다.
- 이때 해당 공통 계층을 사내 플랫폼으로 끌어올려야 합니다. 모든 계층을 한 번에 짓지 않고, 반복 수요가 증명된 계층부터 만들어야 합니다.

즉 "전사 Private AI 플랫폼을 먼저 깔자"가 아니라 "파일럿들이 공통으로 요구하는 것을 사내 플랫폼으로 승격하자"가 됩니다. 이것이 [문서 01](01-diagnosis.md)의 빅뱅 플랫폼 안티패턴을 피하는 길입니다.

## 7.6 역량 플라이휠과 Private AI

[문서 03](03-alternative.md)에서 파일럿마다 축적한 자산은 Private AI 토대 위에서 더 크게 재사용됩니다.

- 파일럿에서 만든 **평가셋**은 사내 모델 선택·튜닝·회귀 검증의 기준이 됩니다.
- **데이터·검색 파이프라인**은 사내 RAG 플랫폼의 출발점이 됩니다.
- **가드레일·프롬프트 패턴**은 플랫폼 공통 정책으로 승격됩니다.
- **접근통제·로깅** 경험은 사내 거버넌스 체계로 굳어집니다.

외부 API로만 실험하고 자산을 남기지 않으면, 막상 Private로 옮길 때 처음부터 다시 시작하게 됩니다. 그래서 외부에서 파일럿하더라도 **재사용 가능한 형태로 자산을 남기는 것**이 전환 비용을 결정합니다.

## 7.7 Private AI를 또 다른 빅뱅으로 만들지 말 것

가장 흔한 함정은 "이제 Private AI를 한다"가 다시 전사 빅뱅 플랫폼 구축으로 변질되는 것입니다. Private AI 자체가 목표가 되면 [문서 01](01-diagnosis.md)의 빅뱅 안티패턴으로 되돌아갑니다.

원칙은 동일하게 유지해야 합니다.

- 검증된 유스케이스가 요구하는 만큼만 갖추어야 합니다.
- 계층은 수요가 증명된 순서로 채워야 합니다.
- 사이징·비용은 [⑥](https://github.com/JaeHoYun/vcf-private-ai-sizing-cost), 보안·거버넌스는 [⑤](https://github.com/JaeHoYun/vcf-private-ai-security-governance), 구축·운영은 [①](https://github.com/JaeHoYun/vcf-private-ai-guide)을 참조해 과투자·과소투자를 피해야 합니다.
- 플랫폼이 커져도 의사결정은 여전히 two-way-door로 구분해야 합니다.

Private AI는 점진적 AX의 토대이지, 점진적 확장 원칙을 면제받는 예외가 아닙니다. 다음 문서는 이 모든 것을 실제 시간축의 실행 계획으로 묶습니다.
